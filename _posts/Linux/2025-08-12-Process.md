`program.c`
```c
int main( int argc, char * argv[] )
{
    int i;
    printf("\nYour program name is %s\n", argv[0]);
    printf("Your arguments are: \n");
    for(i = 1; i < argc; i++)
        printf("\t %s starts with %c\n", argv[i], argv[i][0]);

}
```

## Virtual Memory
Việc ghi trực tiếp vào memory trên MCU thì được nhưng với Linux thì nó không ghi trực tiếp mà nó có kĩ thuật ghi gọi là `virtual memory`.
Hệ điều hành sẽ cấp cho process một khoảng tài nguyên
- `Process virtual address space` hệ điều hành chia thành page nhỏ.

- `Physical Memory`(RAM): cũng được hệ điều hành chia thành nhiều page nhỏ.

- `Page table`: dùng để mapping address trên `virtual memory` với `Physical Memory` làm cho biết với địa chỉ nào trên virtual memory sẽ lưu vào address nào trên physical.

# 4. Operation on Process
## System call fork()
Tạo ra 1 Process con từ Process hiện tại (Parent Process).

![alt text](/assets/Linux/process/fork.png)

```c
#include <unistd.h>
#include <stdio.h>

int main() {
    int x = 1;
    printf("Before fork\n");

    pid_t pid;

    pid = fork(); // fork is called here

    // both processes execute from here
    if (pid < 0) {
        perror("fork failed");
        return 1;
    } else if (pid == 0) {
        // Child process
        x++;
        printf("\n");
        printf("This is the child process. My PID = %d. My parent's PID = %d\n", getpid(), getppid());
        printf("In child process x = %d\n", x);
    } else {
        // Parent process
        printf("\n");
        printf("This is the parent process. My PID = %d. My child PID = %d. My parent's PID = %d\n", getpid(), pid, getppid());
        printf("In parent process x = %d\n", x);
    }

    printf("Both processes continue here.\n");

    return 0;
}
```

**Result**
```c
Before fork

This is the parent process. My PID = 147931. My child PID = 147932. My parent's PID = 74569
In parent process x = 1
Both processes continue here.

This is the child process. My PID = 147932. My parent's PID = 147931
In child process x = 2
Both processes continue here.
```
## 1. `fork()` làm gì?
- `fork()` tạo ra một **tiến trình con (child process)** từ tiến trình cha (parent process).
- Sau khi gọi `fork()`, cả cha và con **cùng chạy tiếp từ vị trí ngay sau lệnh `fork()`**.
- Kết quả trả về từ `fork()`:
  - Trong **cha**: trả về **PID của con** (một số dương).
  - Trong **con**: trả về **0**.
  - Nếu lỗi: trả về `-1`.

Process chilren có thể `fork()` thêm.

## Chạy chương trình khác trong tiến trình con

Sau khi fork(), tiến trình con là bản sao của cha.

Nếu muốn con chạy hẳn một chương trình khác, ta gọi một hàm exec*.

Khi exec chạy thành công, mã nguồn hiện tại trong con bị thay thế, chỉ còn chương trình mới.

| Hàm        | Đặc điểm                                                                 |
| ---------- | ------------------------------------------------------------------------ |
| `execl()`  | Liệt kê tham số theo danh sách (list).                                   |
| `execlp()` | Giống `execl()`, nhưng tự tìm chương trình trong biến môi trường `PATH`. |
| `execv()`  | Truyền tham số qua mảng (vector).                                        |
| `execvp()` | Giống `execv()`, nhưng tìm trong `PATH`.                                 |
| `execve()` | Hệ thống lõi, cho phép truyền cả môi trường (environment).               |

## Tại sao fork() khi gọi fork() chỉ chạy đoạn chương trình sau fork()?
Gọi `fork()` copy virtual memory của parent process tạo thành children process đồng thời copy  **Program Counter** register này trong CPU chứ lệnh chạy tiếp theo nên các lệnh trước `fork()` trong children process không được chạy nữa.
### 1. Quá trình chia sẽ code và data (copy-on-write)
Khi fork() được gọi tiến trình cha và con không được sao chép ngay lập tức mà, ban đầu cả 2 tiến trình cha và con đều dùng chung 1 vùng nhớ cho đến khi 1 trong 2 có sự thay đổi thì mới đưa tiến trình con qua vùng nhớ được copy, đây là cách tiết kiệm tài nguyên hệ thống.
### 2. Trạng thái trả về của fork()
Sau khi gọi fork(), có sự phân biệt rõ ràng giữa tiến trình cha và tiến trình con, dựa vào giá trị trả về của fork():
Tiến trình cha nhận giá trị trả về là PID (process ID) của tiến trình con.
Tiến trình con nhận giá trị trả về là 0
## Tại sao nhận giá trị 0 khi fork ở tiến trình con
Khi tiến trình con nhận giá trị 0, việc phân biệt giữa tiến trình cha và tiến trình con trở nên rất rõ ràng và dễ dàng trong mã nguồn.

# 6. Orphant and Zombie Process
## 1. Tiến trình mồ côi (Orphant Process)
1 tiến trình con mà cha chết gọi là tiến trình mồ côi. Cha return trước con gọi là mồ côi.

![alt text](/assets/Linux/process/orphant.png)

Khi tiến trình cha chết đi thì tiến trình con được cụ tổ quản lý là tiến trình **init** (Init process) là thành phần đầu tiên được gọi trong hệ điều hành.

## 2. Tiến trình zombie (Zombie Process)
Khi 1 Process kết thúc nhưng vẫn còn thông tin trong process table cho hệ điều hành quản lý làm cho tài nguyên trong hệ điều hành đầy.

![alt text](/assets/Linux/process/Z+.png)

**Z+**: Là zombie

## 3. Để dọn dẹp zombie


### Cách 1
![alt text](/assets/Linux/process/wait.png)

dùng wait ra để khi con chết thì hệ điều hành báo. Tránh tạo rác cho hệ thống. 
Với tiến trình zombie không dùng **kill -9** được, thì kill chết cha đi thì thằng con sẽ được cụ tổ quản lý, để cụ tổ có hàm wait quản lý giải phóng hết mấy con.

### Cách 2
Kill parent process thì zombie process sẽ được quản lý bởi Init process, nó sẽ xóa sạch mấy zombie process đó luôn.
